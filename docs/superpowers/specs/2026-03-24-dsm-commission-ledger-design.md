# DSM Commission Ledger — Design Spec

**Date:** 2026-03-24
**Module:** DSM/DSA Commission Tracking
**Owner:** Admin role (not HR — DSMs are external agents, not employees)

---

## Problem

DSMs (Dealer Sales Managers / DSAs) submit leads into the CRM. When those leads convert to closed deals, the DSM should be compensated. Currently there is no way to track what commission is owed, what has been paid, or for the DSM to provide banking details for payment. This creates manual tracking overhead and payment disputes.

## Solution

An append-only commission ledger per DSM with:
- Auto-generated credit entries when DSM-submitted leads reach "Closed Won"
- Admin-recorded debit entries when payments are made
- Running balance showing outstanding amount owed
- DSM self-service banking details and read-only ledger view
- Per-project commission rates configured on the project entity

---

## Data Model

### New Table: `dsmCommissionEntries`

Append-only ledger following the same pattern as `pettyCashEntries`.

| Field | Type | Description |
|-------|------|-------------|
| `dsmUserId` | `id("users")` | The DSM this entry belongs to |
| `type` | `"credit" \| "debit"` | Credit = commission earned, Debit = payment made to DSM |
| `amount` | `number` | Amount in INR (always positive) |
| `leadId` | `optional id("leads")` | Link to the lead that earned the commission (for auto credits) |
| `projectId` | `optional id("projects")` | Project context (for auto credits) |
| `description` | `string` | Human-readable description |
| `isAutoGenerated` | `boolean` | `true` if auto-created on Closed Won, `false` if manually entered |
| `isVoided` | `boolean` | Soft-delete flag |
| `voidedAt` | `optional number` | When voided |
| `voidedBy` | `optional id("users")` | Who voided |
| `voidReason` | `optional string` | Why voided |
| `createdBy` | `id("users")` | Who created (system user for auto, admin for manual) |
| `createdAt` | `number` | Timestamp |

**Indexes:**
- `byDsmUserId` — query all entries for a DSM
- `byDsmUserIdAndType` — query credits or debits for a DSM
- `byLeadId` — check if commission already exists for a lead (prevent duplicates)
- `byCreatedAt` — chronological ordering

### Modified Table: `projects`

Add one field:

| Field | Type | Description |
|-------|------|-------------|
| `dsmCommissionAmount` | `optional number` | Fixed commission amount in INR for this project |

### Modified Table: `users`

Add banking detail fields (used by DSM role only):

| Field | Type | Description |
|-------|------|-------------|
| `bankName` | `optional string` | Bank name |
| `bankAccountNumber` | `optional string` | Account number |
| `bankIfscCode` | `optional string` | IFSC code |
| `bankAccountHolderName` | `optional string` | Account holder name |

## Auto-Generation Flow

When a lead's status changes to **"Closed Won"** (in the existing `updateStatus` mutation in `convex/leads.ts`):

1. Check if `lead.submittedBy` exists AND `lead.source === "dsm"`
2. Query `dsmCommissionEntries` by `leadId` to prevent duplicate commission
3. Look up `project.dsmCommissionAmount` from the lead's project
4. If commission amount is configured and no existing entry:
   - Create a `credit` entry in `dsmCommissionEntries` with:
     - `dsmUserId`: lead.submittedBy
     - `amount`: project.dsmCommissionAmount
     - `leadId`: lead._id
     - `projectId`: lead.projectId
     - `description`: "Commission for {leadName} — {projectName}"
     - `isAutoGenerated`: true
     - `isVoided`: false
     - `createdBy`: the user who changed the status
5. Log activity: `entityType: "dsm_commission"`, `action: "commission_earned"`

If the project has no `dsmCommissionAmount` set, no commission entry is created (silent skip — admin can always add manually later).

## Convex Functions

### New File: `convex/dsmCommissions.ts`

**Queries:**
- `getByDsm({ dsmUserId, paginationOpts })` — Paginated ledger entries for a DSM. DSM role: own entries only. Admin: any DSM.
- `getSummary({ dsmUserId })` — Returns `{ totalCredits, totalDebits, balance }` for a DSM (excluding voided entries).
- `getAllDsmBalances({})` — Admin-only. Returns all DSM users with their `{ totalCredits, totalDebits, balance }`. For the admin overview page.

**Mutations:**
- `recordPayment({ dsmUserId, amount, description })` — Admin creates a debit entry (payment made to DSM).
- `recordManualCredit({ dsmUserId, amount, description, leadId?, projectId? })` — Admin creates a manual credit entry (for edge cases like bonus, adjustment).
- `voidEntry({ entryId, reason })` — Admin voids an entry with a reason. Sets `isVoided`, `voidedAt`, `voidedBy`, `voidReason`.

### Modified File: `convex/leads.ts`

In the `updateStatus` mutation, after the existing "Closed Won" logic (which already triggers after-sales completion), add the auto-commission creation logic described above.

### Modified File: `convex/projects.ts`

Support `dsmCommissionAmount` in project create and update mutations.

### Modified File: `convex/schema.ts`

Add `dsmCommissionEntries` table, new fields on `projects` and `users`.

### New/Modified on Users

Add a `updateBankDetails` mutation allowing DSM users to update their own banking fields. Admin can also view these.

## Pages & UI

### DSM Portal — "My Commissions" (`/dsm/commissions`)

New page accessible from DSM sidebar.

**Layout:**
1. **Summary Cards** (top row):
   - Total Earned (sum of non-voided credits)
   - Total Received (sum of non-voided debits)
   - Outstanding Balance (credits - debits)

2. **Ledger Table** (main content):
   - Columns: Date, Description, Credit (green), Debit (red), Balance
   - Voided entries shown with strikethrough and "Voided" badge
   - Paginated (25 per page)
   - Most recent first

3. **Bank Details Section** (below or as a tab):
   - Editable form: Bank Name, Account Number, IFSC Code, Account Holder Name
   - Save button with success toast
   - Shows "No banking details added" prompt if empty

### Admin — DSM Commissions (`/admin/commissions`)

New admin page accessible from admin sidebar.

**Layout:**
1. **DSM Overview Table** (default view):
   - Columns: DSM Name, Total Earned, Total Paid, Outstanding Balance, Actions
   - Click row → navigates to individual DSM ledger
   - "Record Payment" quick action button per row

2. **Individual DSM Ledger** (`/admin/commissions/[dsmUserId]`):
   - Same summary cards + ledger table as DSM portal view
   - Plus: DSM's bank details (read-only display)
   - Plus: "Record Payment" button → dialog with amount + description
   - Plus: "Add Manual Credit" button → dialog with amount + description + optional lead/project
   - Plus: "Void" action on each entry → dialog with reason
   - Activity log section at bottom

### Project Form Updates

In the project create/edit form (`components/projects/project-create-dialog.tsx` and admin project edit page):
- Add "DSM Commission (INR)" number input field
- Optional — if not set, no auto-commission for this project

### DSM Sidebar Update

Add "My Commissions" nav item to DSM sidebar (`components/layout/app-sidebar.tsx`).

|--------|-------|-----------|--------|
| View ledger | All DSMs | Own only | No |
| Record payment (debit) | Yes | No | No |
| Record manual credit | Yes | No | No |
| Void entry | Yes | No | No |
| Update bank details | View only | Own only | No |
| Set project commission rate | Yes | No | No |

## Edge Cases

1. **Project has no commission rate**: No auto-entry created. Admin can add manual credit later.
2. **Lead already has commission entry**: Duplicate check by `leadId` prevents double-crediting (e.g., if status oscillates).
3. **Lead moves back from Closed Won**: Commission entry remains. Admin can void if needed — this is intentional (append-only ledger, no auto-void).
4. **DSM account deactivated**: Ledger entries persist. Outstanding balance still visible to admin.
5. **Multiple leads from same DSM on same project**: Each gets its own commission entry.
6. **Admin records overpayment**: Balance goes negative (paid more than earned). Visible in UI — admin's responsibility to reconcile.

## Activity Logging

All commission actions logged via existing `activityLogger`:
- `entityType: "dsm_commission"`
- Actions: `commission_earned`, `payment_recorded`, `manual_credit_added`, `entry_voided`
- Details: JSON with amount, DSM name, lead name (where applicable)
